## Key Patterns

**State management:** Every feature has a `*Notifier extends _$*Notifier` (Riverpod code-gen). State is `AsyncValue<T>`. Screens use `ref.listen()` to react to state changes (navigate on success, show SnackBar on error). `ref.watch()` drives loading spinners.

**Provider pattern:**
```dart
// provider file
@riverpod
class LoginNotifier extends _$LoginNotifier {
  @override
  AsyncValue<void> build() => const AsyncData(null);

  Future<void> login(String email, String password) async {
    state = const AsyncLoading();
    state = await AsyncValue.guard(() => ref.read(authRepositoryProvider).login(email, password));
  }
}

// screen — listen for side effects, watch for UI state
ref.listen(loginNotifierProvider, (_, next) {
  next.whenOrNull(
    error: (e, _) => ScaffoldMessenger.of(context).showSnackBar(...),
    data: (_) => context.go('/dashboard'),
  );
});
final isLoading = ref.watch(loginNotifierProvider).isLoading;
```

**Repositories:** All Firebase calls live in `data/repositories/`. They throw exceptions on failure — `AsyncValue.guard()` catches them. Providers never touch Firebase directly. All repository providers use `keepAlive: true`.

**After adding/changing any `@riverpod` annotation**, run:
```bash
dart run build_runner build --delete-conflicting-outputs
```

**Auth redirect:** GoRouter `redirect` callback checks `FirebaseAuth.instance.currentUser`. Unauthenticated users → `/login`; authenticated users hitting auth routes → `/dashboard`.

## Firebase Data Model (Firestore)

```
users/{uid}          → { uid, fullName, email, department }
studyHalls/{id}      → { name, location, capacity, tableCount, imageUrl }
bookings/{id}        → { userId, hallId, hallName, tableNumber, seatNumber,
                         date (ISO yyyy-MM-dd), timeSlotLabel, startHour,
                         durationMinutes, createdAt, status ("active"|"cancelled") }
bookedSlots/{key}    → { bookingId, createdAt }   ← slot-lock mutex, key format:
                         {hallId}_t{table}_s{seat}_{date}_h{hour}
```

**Firestore indexes needed:** `bookings` composite index on `(hallId, tableNumber, date, status)` and `(userId, createdAt DESC)`.

**Slot-lock pattern:** `createBooking()` and `cancelBooking()` both use `db.runTransaction()` to atomically read the `bookedSlots` lock document and write the booking — preventing double-booking under concurrent requests.
